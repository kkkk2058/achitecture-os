## CH 3. 데이터

### 1. 핵심 개념 3개

- 정보 단위: 비트는 0과 1을 표현하는 가장 작은 정보 단위

### 2. 내 말로 설명하기

0과 1을 표현하는 가장 작은 정보 단위는 비트이다. 1바이트는 8비트이다. 워드는 cpu가 한 번에 처리할 수 있는 크기 단위이다. 이진법을 활요해서 음수도 표현할 수 있는데 2의 보수를 활용하여 계산한다. 문자 집합은 컴퓨터가 이해할 수 있는 문자들이고 문자를 0과1로 변환하는 인코딩 반대의 과정은 디코딩이다. 한글 인코딩은 한글의 특징을 고려해서 만들어졌다. 현재는 utf-8을 주로 활용한다.

### 3. 헷갈려서 AI에게 확인한 부분

- AI 답변을 보고 이해한 내용: 0의 유일성, 뺄셈의 가산화가 가능하다

### 4. 개발 또는 일상과 연결해 본 예시

모델 경량화와 정보 단위: LLM와 같은 대형 모델을 다룰 때, 가중치를 FP16(16비트)에서 INT8(8비트) 또는 INT4(4비트)로 양자화(Quantization)하는 작업은 결국 정보 단위를 줄여 메모리 사용량을 절감하는 과정이다. 16비트 데이터를 4비트로 줄이면 이론적으로 메모리 사용량이 1/4로 감소하여 로컬 GPU 환경에서 추론이 가능하도록 만들 수 있다.

센서 데이터와 이진법: XR 기기의 IMU 센서나 포인트 클라우드 좌표 데이터는 정밀도에 따라 비트 수가 결정된다. 만약 가속도 값을 8비트로 표현한다면 -128부터 127까지만 표현 가능하므로, 급격한 움직임 발생 시 데이터가 허용 범위를 넘어서는 오버플로우(Overflow)가 발생해 경로 추적에 오류가 생길 수 있다.

멀티모달 로그와 UTF-8: Whisper를 통해 추출한 음성 텍스트나 BLIP으로 생성한 이미지 캡셔닝 결과를 JSON으로 저장할 때, 인코딩을 utf-8로 설정하지 않으면 한글 로그가 깨지는 현상이 발생한다.

### 2. 내 말로 설명하기

오퍼랜드라는 공간을 할당하여 실제로 연산에 사용할 데이터와 데이터의 주소를 저장한다. 나중에 주소를 찾아서 실제 내가 사용하고 싶은 데이터를 찾아갈 수 있다. 이 주소를 유효 주소라고 한다. 즉시 주소, 간접 주소 등등 여러 지정 방식들이 존재하며 각각 접근 속도와 데이터 크기의 trade-off 가 존재한다.

### 3. 헷갈려서 AI에게 확인한 부분

- 처음 헷갈렸던 내용: 명령어 주소 지정 방식이 다양하게 있는 이유
- AI에게 물어본 질문: 명령어 주소 지정 방식이 다양하게 있는 이유
- AI 답변을 보고 이해한 내용: 오퍼랜드 필드의 비트 수가 제한되어있기 때문이다.(공간의 한계) 간접 주소 지정이나 레지스터 간접 지정을 사용하면 작은 비트 수로도 수십 GB의 메모리 영역 어디든 갈 수 있기 때문이다.
- 아직 남은 질문: 실제로 어떤 상황일 때 이 지정방식이 달라질까?
  - 1. 상수 초기화 및 리터럴 사용 (즉시 주소 지정) 코드에서 변하지 않는 고정된 숫자를 사용할 때 발생합니다.
    상황: int count = 10; 또는 x = x + 1;
    작동: 숫자 10이나 1은 굳이 메모리에 저장했다가 가져올 필요가 없다. 명령어의 오퍼랜드 필드에 숫자를 직접 박아 넣는다.

    상황: for(int i=0; i<1000; i++)에서 i의 값 변화.
    상황: int count = 10; 또는 x = x + 1;
    작동: 컴파일러는 i를 메모리에 매번 쓰지 않고, CPU 내부의 레지스터에 할당하여 연산한다.
